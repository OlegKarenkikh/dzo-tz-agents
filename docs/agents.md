# Описание агентов

## Агент ДЗО — Инспектор заявок дочерних обществ

### Назначение

Агент автоматически проверяет входящие заявки от дочерних обществ (ДЗО) на полноту и корректность перед регистрацией в системе ЭДО «Тезис». Работает через электронную почту: получает письма с вложениями, анализирует их с помощью LLM и отправляет ответ.

### Файлы

- `agent1_dzo_inspector/agent.py` — создание LangChain AgentExecutor
- `agent1_dzo_inspector/runner.py` — оркестратор обработки писем
- `agent1_dzo_inspector/tools.py` — инструменты (функции, вызываемые LLM)

### SLA

| Операция | Срок |
|----------|------|
| Реакция на входящее письмо | 2 часа |
| Запрос недостающих данных | 1 час |
| Эскалация при отсутствии ответа от ДЗО | 2 дня |

### Чек-лист №1: Комплектность вложений

| № | Критерий | Обязательный |
|---|----------|:---:|
| 1.1 | Наличие файла ТЗ | ✅ |
| 1.2 | Наличие спецификации (для сложных закупок) | ⚠️ |
| 1.3 | Файлы открываются, не защищены паролем | ✅ |

### Чек-лист №2: Обязательные реквизиты

| № | Реквизит | Примечание |
|---|----------|-----------|
| 2.1 | Наименование закупки | Точное наименование |
| 2.2 | Количество с единицами измерения | Например: 5 шт., 100 кг |
| 2.3 | Желаемый срок поставки | Конкретная дата (дд.мм.гггг) |
| 2.4 | Инициатор — ФИО и контакты | Телефон или email |
| 2.5 | Место поставки | Точный адрес |

### Чек-лист №3: Дополнительные поля

| № | Поле |
|---|------|
| 3.1 | Бюджет в рублях (с НДС или без) |
| 3.2 | Предмет закупки |
| 3.3 | Обоснование закупки |
| 3.4 | Желаемая дата поставки |
| 3.5 | Рекомендуемые поставщики (ИНН) |

### Возможные решения

| Решение | Условие | Действие |
|---------|---------|---------|
| ✅ **Заявка полная** | Все обязательные реквизиты заполнены | Формируется форма Тезис, отправляется подтверждение |
| ⚠️ **Требуется доработка** | Отсутствуют один или несколько обязательных полей | Отправляется запрос на дополнение с проектом исправленной заявки |
| 🔴 **Требуется эскалация** | Критические противоречия или нестандартная ситуация | Уведомление руководителю |

### Инструменты (LangChain tools)

| Инструмент | Описание | Вход | Выход |
|-----------|----------|------|-------|
| `generate_validation_report` | Формирует структурированный отчёт по чек-листам | `decision`, `checklist_*`, `missing_fields` | JSON с `stats`, `missing_fields` |
| `generate_tezis_form` | Создаёт HTML-форму для системы «Тезис» | Реквизиты закупки | HTML с формой |
| `generate_info_request` | Письмо-запрос недостающих данных | `dzo_name`, `missing_fields` | HTML-письмо + `subject` |
| `generate_escalation` | Уведомление об эскалации | `subject`, `reason`, `details` | HTML для менеджера |
| `generate_response_email` | Итоговое письмо отправителю | `decision`, `subject`, `agent_summary` | HTML-письмо |
| `generate_corrected_application` | Проект исправленной заявки | Поля с изменениями | HTML-документ |
| `analyze_tz_with_agent` | Делегирует анализ ТЗ агенту ТЗ и возвращает сводку | `tz_text`, `email_subject`, `source_sender` | JSON с `tzAgentAnalysis` |
| `invoke_peer_agent` | Универсальный вызов любого другого агента | `target_agent`, `query_text`, контекст | JSON с `peerAgentResult` |

### Межагентные вызовы (agent-as-tool)

В проекте поддерживается универсальный bridge `shared/agent_tooling.py`, который позволяет
одному агенту вызывать другого как tool c контролем маршрутов.

- Реестр фабрик агентов настраивается через `AGENT_TOOL_REGISTRY` (JSON).
- Разрешённые маршруты настраиваются через `AGENT_TOOL_PERMISSIONS` (JSON).
- Глобальный флаг включения: `AGENT_TOOL_ENABLED=true/false`.
- По умолчанию действует политика `all_except_self`: агент может вызывать любой другой агент из реестра.
- Новые агенты, созданные по naming-convention `agentN_<id>_inspector` + `create_<id>_agent`,
  автоматически обнаруживаются и становятся доступны для межагентных вызовов.

Пример:

```bash
AGENT_TOOL_ENABLED=true
AGENT_TOOL_REGISTRY={"dzo":"agent1_dzo_inspector.agent:create_dzo_agent","tz":"agent2_tz_inspector.agent:create_tz_agent"}
AGENT_TOOL_PERMISSIONS={"*":["*"]}
```

### Пример входных данных

```
📧 ВХОДЯЩАЯ ЗАЯВКА ОТ ДЗО
═══════════════════════════════════════════
От: dzo_company@example.com

── ТЕЛО ПИСЬМА ──
```

## Агент ТЗ — Инспектор технических заданий

### Назначение

Агент проверяет технические задания на соответствие стандартам и полноту разделов перед закупочной процедурой. Принимает документы через почту или REST API, анализирует структуру ТЗ и выдаёт заключение.

### Файлы

- `agent2_tz_inspector/agent.py` — создание LangChain AgentExecutor
- `agent2_tz_inspector/runner.py` — оркестратор обработки писем
- `agent2_tz_inspector/tools.py` — инструменты агента

### Чек-лист разделов ТЗ

| № | Раздел | Описание |
|---|--------|----------|
| 1 | Цель и задачи | Цель закупки, задачи, ожидаемые результаты |
| 2 | Технические требования | Характеристики, параметры, стандарты |
| 3 | Функциональные требования | Функции, режимы работы |
| 4 | Требования к качеству | Стандарты качества, сертификаты |
| 5 | Объём и сроки | Количество, единицы измерения, даты |
| 6 | Место поставки | Точный адрес, условия доставки |
| 7 | Порядок приёмки | Процедура проверки и приёмки |
| 8 | Гарантийные обязательства | Срок гарантии, условия обслуживания |

### Возможные решения

| Решение | Условие | Действие |
|---------|---------|---------|
| ✅ **Соответствует** | Все разделы присутствуют и заполнены | Отправляется подтверждение |
| ⚠️ **Требует доработки** | Некоторые разделы отсутствуют или неполные | Отправляется исправленный вариант ТЗ |
| 🔴 **Не соответствует** | Критические нарушения требований | Отправляется подробный отчёт о несоответствии |

### Инструменты (LangChain tools)

| Инструмент | Описание | Вход | Выход |
|-----------|----------|------|-------|
| `generate_json_report` | Структурированный JSON-отчёт по разделам | `overall_status`, `sections`, `critical_issues` | JSON со статистикой |
| `generate_corrected_tz` | HTML-документ с исправленным ТЗ | `title`, `original_sections`, `modifications` | HTML с выделением изменений |
| `generate_email_to_dzo` | Письмо ДЗО с результатом проверки | `decision`, `issues`, `recommendations` | HTML-письмо |
| `invoke_peer_agent` | Универсальный вызов любого другого агента | `target_agent`, `query_text`, контекст | JSON с `peerAgentResult` |

### Пример входных данных

```
📧 МЕТАДАННЫЕ ПИСЬМА:
От: tz_company@example.com
Тема: Техническое задание на поставку серверов
Тело письма: Направляем ТЗ на согласование.

── ВЛОЖЕНИЯ ──
──── Файл: техзадание.pdf ────
Техническое задание на поставку серверного оборудования

1. Цель закупки
Приобретение серверов для нужд ЦОД компании.

2. Технические требования
- CPU: Intel Xeon, не менее 16 ядер
- RAM: не менее 256 ГБ ECC
- Storage: 4x 1.92 TB NVMe SSD
...
```

### Пример результата (решение: Требует доработки)

```json
{
    "overall_status": "Требует доработки",
    "stats": {"ok": 5, "issues": 3, "total": 8},
    "critical_issues": ["Отсутствует раздел 7 (порядок приёмки)"],
    "recommendations": ["Добавьте раздел о гарантийных обязательствах"]
```

## Агент Тендер (agent21) — Парсер тендерной документации

> **Примечание о нумерации:** `agent21` — вспомогательный (peer) агент для `agent1`. Цифра `21` означает «агент 2, подчинённый агенту 1». Он не является самостоятельным агентом в последовательности обработки, а вызывается по запросу через `invoke_peer_agent` из agent1 или напрямую через `runner.py` для пакетной обработки документов.

### Назначение

Агент извлекает из закупочной документации полный перечень документов,
которые должен предоставить участник закупки, с привязкой к разделам и
формулировками требований.

Поддерживает типовые сценарии для закупок по 44-ФЗ и 223-ФЗ, а также
аналогичные корпоративные регламенты.

### Файлы

- `agent21_tender_inspector/agent.py` — создание LangGraph-агента
- `agent21_tender_inspector/runner.py` — оркестратор обработки документов (пакетный режим, без email-polling)
- `agent21_tender_inspector/tools.py` — инструменты извлечения структуры требований

### Выходные артефакты

| Артефакт | Назначение |
|---|---|
| `output` | Текстовый результат агента |
| `decision` | `documents_found` или `tool_error` |
| `trace_json` (при наличии) | Трассировка шагов и источников |

### Инструменты (LangChain tools)

| Инструмент | Описание | Вход | Выход |
|---|---|---|---|
| `generate_document_list` | Извлекает структурированный перечень требуемых документов | `query` (JSON-строка) | JSON-список документов |
| `invoke_peer_agent` | Универсальный вызов любого другого агента | `query` (JSON-строка с `target_agent`, `query_text`) | JSON с `peerAgentResult` |

### Основные риски и контроль качества

| Риск | Как контролируется |
|---|---|
| Пропуск обязательного документа | Проверка по ключевым паттернам и повторный проход по разделам |
| Некорректная ссылка на раздел | Сохранение цитаты и идентификатора блока/раздела |
| Смешение требований заказчика и участника | Нормализация формулировок и фильтр на роль субъекта |

### Рекомендуемые входные данные

- Полный текст документации закупки или вложение (PDF/DOCX/XLSX)
- Тема/контекст закупки в поле `subject`
- Имя файла и метаданные отправителя для дедупликации и аудита

### Пример результата

```json
{
    "decision": "documents_found",
    "documents": [
        {
            "name": "Выписка из ЕГРЮЛ",
            "section": "Раздел 3. Требования к участникам",
            "requirement": "Не старше 6 месяцев"
        },
        {
            "name": "Декларация о соответствии",
            "section": "Приложение 2",
            "requirement": "По форме заказчика"
        }
    ]
}
```

## Агент Collector (agent3) — Сборщик документов тендерного отбора

### Назначение

Агент автоматизирует сбор и проверку документов от участников тендерного отбора (ТО) для Управления централизованных закупок (УЦЗ). Получает список писем и участников, фильтрует по номеру ТО, сопоставляет отправителей с участниками, классифицирует вложения (анкета / NDA / прочее), валидирует данные и формирует отчёт.

### Файлы

- `agent3_collector_inspector/agent.py` — создание LangGraph-агента
- `agent3_collector_inspector/runner.py` — пакетная обработка JSON-файлов с данными ТО, CLI
- `agent3_collector_inspector/tools.py` — инструменты сбора и валидации документов

### Алгоритм работы

1. Получить данные: `tender_id`, список писем (`emails`), список участников (`participants_list`)
2. Отфильтровать письма по номеру ТО в теме
3. Для каждого письма: идентифицировать участника (по email / ИНН / наименованию), классифицировать вложения, проверить ИНН и наименование
4. Сформировать структуру папок
5. Вызвать `collect_tender_documents` с результатами анализа
6. При необходимости вызвать `invoke_peer_agent` для смежной проверки

### Инструменты (LangChain tools)

| Инструмент | Описание | Вход | Выход |
|-----------|----------|------|-------|
| `collect_tender_documents` | Сбор и валидация документов участников ТО | JSON: `tender_id`, `emails`, `participants_list` | JSON-отчёт: участники, расхождения, структура папок |
| `invoke_peer_agent` | Универсальный вызов любого другого агента | JSON: `target_agent`, `query_text` | JSON с `peerAgentResult` |

### Валидация

| Проверка | Критичность |
|----------|-------------|
| Совпадение ИНН из анкеты со сводной таблицей | Высокая |
| Совпадение наименования организации | Средняя |
| Наличие анкеты у каждого участника | Высокая |
| Наличие NDA у каждого участника | Высокая |

### Пример результата

```json
{
    "tender_id": "3115-ДИТ-Сервер",
    "total_expected_participants": 5,
    "received_count": 3,
    "missing_count": 2,
    "participants": [
        {
            "name": "АО «Ромашка»",
            "status": "received",
            "anketa_received": true,
            "nda_received": true,
            "inn_match": true,
            "discrepancies": []
        }
    ],
    "discrepancies": []
}
```

## Общие модули

### `shared/file_extractor.py`

Поддерживаемые форматы:

| Формат | Метод извлечения | Fallback |
|--------|-----------------|---------|
| PDF | pdfplumber | GPT-4o Vision OCR |
| DOCX | python-docx | GPT-4o Vision OCR |
| DOC | — | GPT-4o Vision OCR |
| XLSX | openpyxl → Markdown-таблица | — |
| XLS | xlrd → Markdown-таблица | — |
| JPG/PNG/BMP/... | GPT-4o Vision OCR | — |

### `shared/email_client.py`

Получает UNSEEN письма из IMAP-ящика:
- Декодирует заголовки (RFC 2047, кириллица)
- Извлекает тело письма (text/plain или text/html)
- Собирает вложения с байтами и base64-кодировкой
- Помечает письма как прочитанные (`\Seen`)
- Обрабатывает ошибки соединения (возвращает `[]`, не бросает исключение)
