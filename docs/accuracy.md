# Точность агентов: измеренные метрики

## Методология

- **Ground truth:** экспертное решение специалиста по госзакупкам
- **Тестовые документы:** реальные публичные источники (ЕЭК, rbank.by, zakupki.gov.ru) и синтетические шаблоны (44-ФЗ/223-ФЗ)
- **Агенты:** tz (Техническое задание), dzo (Заявка ДЗО), tender (Тендерная документация), collector (Сбор документов)
- **Режимы тестирования:**
  - Rules-engine (без LLM): структурный парсинг + keyword matching
  - LLM E2E: полный цикл через API с реальной моделью (qwen3-32b)

## Целевые показатели

| Режим | Документов | Реальных | Агентов | Текущая точность | Целевая точность |
|---|---|---|---|---|---|
| Rules-engine (без LLM) | 10 | 5 | 4 | Измеряется | ≥ 70% |
| LLM E2E (qwen3-32b) | 10 | 5 | 4 | Измеряется | ≥ 85% |

> **Примечание:** Целевая точность (≥ 85%) — доля документов в тестовом реестре, по которым
> решение агента совпало с экспертным (decisions_match). Это отличается от внутреннего порога
> агента: например, агент ТЗ использует score ≥ 95% → «Соответствует», 75–95% → «Принять с замечанием»,
> < 75% → «Вернуть на доработку». Внутренний порог определяет решение по одному документу,
> а целевая точность — долю правильных решений по всему реестру.

**Цель по документам:** ≥ 5 реальных документов из публичных источников (zakupki.gov.ru, ЕЭК, rbank)

## Известные ограничения rules-engine

1. **Ложноположительный результат для rbank:** keyword `г.` встречается в характеристиках оборудования (размер дисплея "23.8\""), а не в поле адреса. Rules-engine неверно определяет наличие раздела «место поставки».
2. **Строгий порог агента vs эксперт:** Агент использует порог 85% для принятия, эксперт допускает принятие с устным замечанием при 87.5% (ЕЭК).
3. **LLM-режим устраняет оба ограничения** за счёт семантического анализа контекста.

## Запуск тестов точности

```bash
# Rules-engine only (быстро, без LLM)
pytest tests/test_real_procurement_docs.py -v

# E2E с LLM (требует API-ключ и работающий сервер)
QWEN_API_KEY=your_key pytest tests/ -m e2e -v
```

## Файлы с тестовыми данными

- `tests/fixtures/real_procurement_docs.py` — тексты документов и ground truth
- `tests/fixtures/real_procurement_expected.json` — структурированные ожидаемые результаты

## Как добавить новый тестовый документ

1. Добавьте текст документа как константу в `tests/fixtures/real_procurement_docs.py`
   - Для тендеров: добавьте `.md` файл в `tests/fixtures/tenders/` и загрузите через `_read_tender()`
   - Для collector: используйте `json.dumps()` для создания JSON-строки
2. Добавьте запись в `REAL_DOCS_REGISTRY` с полями:
   - `agent`: `"tz"`, `"dzo"`, `"tender"` или `"collector"`
   - `text`: ваша константа
   - `filename`: имя файла для отправки в API
   - `subject`: краткое описание
   - `expected`: словарь с `expert_decision`, `key_missing`, `structural_score_pct`
   - `source_url`: ссылка на источник или `None` для синтетических
3. Добавьте запись в `tests/fixtures/real_procurement_expected.json`
4. Запустите `pytest tests/test_real_procurement_docs.py -v` — новый документ автоматически подхватится параметризованным тестом
